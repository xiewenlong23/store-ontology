# 新业务场景接入手册（标准流程）

> **前提**：内核多 vertical 改造已完成（见 `01`）。本手册之后，新增 vertical **零改内核**，纯增量。
> **配套**：模板见 `templates/`（填法见 `03-模板说明.md`）；完整例子见 `04-worked-example-设备维修工单.md`。
> **依据**：`docs/业务本体建模规范.md`（下称"建模规范"）

---

## 0. 接入前的设计准备

动手前先回答（对照建模规范 §1.2、§3.4、§5.2）：
1. 这组业务实体有**独立生命周期 + 受治理写操作 + 多步工作流**吗？是 → 适合建 vertical。否 → 用内核 CRUD 工具即可，不必建 vertical（见 00 §4）。
2. 工作流主对象是什么？（如 RepairTicket）—— 它决定 `workflow_object_type`。
3. 哪些实体是**受治理**的（`edits_only_via_actions: true`）？通常是工作流对象 + 它操作的标的物。
4. 状态机的合法迁移图是什么？画出来。
5. 有没有类似折扣的"数值规则"？若有，定单一事实源文件 + 计算函数。

## Phase A：本体建模（Object / Link）

**做什么**：填 `templates/ontology.ttl.template` → 产出 `<vertical_name>.ttl`。
**放在**：`backend/verticals/<name>/ontology/<name>.ttl`

步骤：
1. 列出所有 Object Type，每个填 `{{prefix}}:{{ClassName}} a rdfs:Class` 块（含 `properties` / `storage` / `status`；受治理实体加 `edits_only_via_actions "true"`）。
2. 列出所有 Link Type，每个填 `rdfs:Property` 块（domain/range/via）。
3. **核对建模规范 §9 检查清单**（重点）：
   - [ ] 命名：Object PascalCase 单数；Link snake_case；每个资源有中英文 label。
   - [ ] 主键 `id`（string）。
   - [ ] 属性类型取自白名单（§6.1），无 string 存日期/数值。
   - [ ] 软引用字段 `{type}_id` 形式，与 Link 的 via 一致。
   - [ ] **via 归属**（§4.3，最易错）：via 字段确实属于 domain 或 range；via 在 range 侧时 comment 写明。
   - [ ] 单位口径（§6.2）：百分比=减扣百分比 int、金额=元、数量带单位。
   - [ ] 枚举属性对应代码侧 Enum。

**验证**：本阶段无代码，用肉眼 + 检查清单。或先写一个最小 parser 测试确认能解析（参考 `backend/tests/test_parser.py`）。

---

## Phase B：Action 契约

**做什么**：每个 Action 填 `templates/action.yaml.template` → 产出 `ontology/actions/<api_name>.yaml`。
**放在**：`backend/verticals/<name>/ontology/actions/`

步骤：
1. 每个 Action 一个 YAML。必填：`api_name` / `target_object_type` / `edits_object_types` / `parameters`（带 constraint）/ `side_effects`。
2. **填 locator_field**（新机制，见 01）：target 是工作流对象时，填定位它的参数名（如 `ticket_id`）；target 是其它对象（如 deduct_stock 的 NearExpiryProduct）填 `target_id` 或不填（默认）。
3. submission_criteria：roles 白名单 + conditions（field 用 `target.status` 或 `<workflow_object_lower>.status`）。
4. side_effects 声明式：create_object / update_object（fields 或 transforms）/ state_transition（from/to）/ notification / external_call。

**核对**（建模规范 §5、§9）：
- [ ] Action 边界正确（受治理写走 Action，读不走，非业务数据走降级 CRUD）。
- [ ] target_object_type 与 locator_field 匹配（target 是 Task 类填 task_id 类，target 是标的物填 target_id）。
- [ ] parameters 带声明式 constraint，数值参数注明单位。
- [ ] side_effects 声明式，不在执行器硬编码。
- [ ] 多 Object 写入保证原子性（executor 用 Repository 原子写）。

---

## Phase C：状态机（若有工作流）

**做什么**：填 `templates/state_machine.py.template` → 产出 `state_machine.py`。
**放在**：`backend/verticals/<name>/state_machine.py`

步骤：
1. 定义 `{{WORKFLOW_OBJECT}}_TRANSITIONS` dict（键=当前状态，值=允许的下一状态列表）+ `TERMINAL_STATES` 集合。
2. config.py 会 import 这两个并填入 `VerticalConfig.state_transitions` / `terminal_states`（见 Phase F）。
3. executor 的 state_transition 副作用自动查这张表校验。

**核对**（建模规范 §5.5、§8 反模式 7）：
- [ ] 所有状态迁移只能由 Action 触发，无直接改 status 的口子。
- [ ] 终态集合正确（终态不可再迁移）。

**无工作流的 vertical 跳过本步**：config 里 `workflow_object_type=None`，不填 state_transitions。

## Phase D：种子数据

**做什么**：每个 Object Type 填 `templates/seed-object.json.template` → 产出 `<storage>.json`。
**放在**：`data/<vertical_name>/<storage>.json`（独立子目录，见 00 §3.3）

步骤：
1. TTL 里声明了哪些 `storage` 文件，就建哪些 JSON（数组）。
2. 填实例：覆盖关键状态值（如起点态 / 中途态 / 终态 / 被拦截态），便于测试。
3. 外键自洽：`{type}_id` 字段指向已存在的实例。

**验证脚本**（对照 clearance 的做法）：
```bash
cd backend && /opt/miniconda3/envs/store-ontology/bin/python -c "
import sys, json; sys.path.insert(0,'.')
# 1. schema 校验（若有 schemas.py）
# 2. FK 完整性：每个 _id 字段指向存在的实例
# 3. 工作流对象 ↔ 标的物状态一致性（如 task in_progress → ticket repairing）
# 4. 库存/计数自洽（若有数量字段）
print('seed 校验通过')
"
```

## Phase E：vertical 专属工具与 Skill

### E.1 专属工具（可选）

若有内核 `query_entity` 满足不了的查询（如需 join 多表 + 算派生字段），填 `templates/vertical-tools.py.template` → 产出 `tools.py`，导出 `TOOLS = [...]`。
**放在**：`backend/verticals/<name>/tools.py`

无专属工具的 vertical 跳过：config 的 `tools_module=None`，纯靠内核 `query_entity`。

### E.2 Skill 文档

填两个模板：
- `templates/skill-knowledge.md.template` → `<name>/skills/<vertical>-knowledge/SKILL.md`（领域知识类）
- `templates/skill-workflow.md.template` → `<name>/skills/<vertical>-workflow/SKILL.md`（工作流编排类）

**核对**（建模规范 §8 反模式 5）：
- [ ] SKILL.md 不引用不存在的实体/工具（最容易遗漏，review 时逐条核对）。
- [ ] 数值规则不重复定义（指向单一事实源文件）。

## Phase F：注册与接线

**做什么**：填 `templates/vertical-config.py.template` → 产出 `config.py`。
**放在**：`backend/verticals/<name>/config.py`

步骤：
1. 构造 `VerticalConfig`，填全路径（ttl/actions/data/skills）+ 工作流字段 + tools_module。
2. 调 `register_vertical(...)`。
3. **确保目录结构**：
   ```
   backend/verticals/<name>/
   ├── __init__.py            # 空文件
   ├── config.py              # 本步产物
   ├── ontology/
   │   ├── <name>.ttl         # Phase A
   │   └── actions/*.yaml     # Phase B
   ├── state_machine.py       # Phase C（无工作流可省）
   ├── tools.py               # Phase E.1（可省）
   └── skills/
       ├── <name>-knowledge/SKILL.md   # Phase E.2
       └── <name>-workflow/SKILL.md
   data/<name>/*.json         # Phase D
   ```
4. **重启服务**：`bootstrap()` 自动发现 `verticals/<name>/config.py` 并注册。**无需改任何内核文件。**

**验证**：
```bash
cd backend && /opt/miniconda3/envs/store-ontology/bin/python -c "
import sys; sys.path.insert(0,'.')
import main
from ontology.vertical import all_verticals
print('verticals:', [c.name for c in all_verticals()])
print('tools:', [getattr(t,'name','') for t in main.tools])
print('skills:', main._skill_paths)
"
```
应看到新 vertical 在列表里，专属工具被聚合，skill 路径收录。

## 常见坑

1. **locator_field 填错**：target 是工作流对象（Task/RepairTicket）填其 id 参数（task_id/ticket_id）；target 是标的物（NearExpiryProduct/Equipment）填 target_id。填反 → target 查不到 → submission 条件 `target.status` 解析失败。
2. **via 归属反了**：见建模规范 §4.3，最易错。Link 的 via 字段必须在 domain 或 range 的属性列表里。
3. **TTL 块结尾**：每个 Class/Property 块必须以 ` .` 行结尾（parser 按行匹配）。
4. **storage 文件名不匹配**：TTL 的 `storage "x.json"` 必须对应 `data/<vertical>/x.json`，否则 Repository 返回空。
5. **Skill 引用幽灵工具**：`allowed_tools` 列的工具必须真实存在（含内核工具 + 本 vertical 工具）。
6. **状态机终态漏了**：终态必须列入 `terminal_states`，否则能从终态再迁移。
