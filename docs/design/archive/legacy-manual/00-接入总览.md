## 0. 一句话定位

**OntologyAgent = 本体驱动的通用 AI Agent 平台。** 通用内核（存储 / 权限 / 审计 / Agent harness / Tool-Skill 体系）固定不变；每个业务场景作为一个 **vertical**，在本体之上声明式建模即可接入，**新增 vertical 零改内核**。

一个 vertical = 一组自洽的 **Object/Link/Action + 工作流(可选) + Skill + 种子数据**。临期出清（clearance）是第一个 vertical；本文档教你接入第二个、第三个。

## 1. kernel / vertical 边界

下表是接入新 vertical 时"复用什么 / 新增什么 / 哪些是内核泄漏需先改"的速查。详细 file:line 见 `01-内核多vertical改造.md`。

| 文件 / 目录 | 性质 | 新 vertical 怎么对待 |
|---|---|---|
| `backend/ontology/parser.py` | **内核** | 复用。vertical 无关（prefix 动态读、prompt intro 来自 config） |
| `backend/ontology/repository.py` | **内核** | 复用。多租户 / 锁 / 原子写 / edits-only |
| `backend/ontology/executor.py` | **内核** | 复用。声明式瘦路由器，target 定位数据驱动 |
| `backend/ontology/action_loader.py` | **内核** | 复用。YAML → ActionDefinition |
| `backend/ontology/state_machine.py` | **内核** | 复用。`is_valid_transition(transitions, terminals)` 接受 per-vertical 表 |
| `backend/ontology/preview_cache.py` | **内核** | 复用。preview→confirm 闭环 |
| `backend/ontology/errors.py` | **内核** | 复用。通用异常 |
| `backend/ontology/vertical.py` | **内核** | 复用。`VerticalConfig` + 注册表 |
| `backend/ontology/bootstrap.py` | **内核** | 复用。自动发现 `verticals/*/config.py` |
| `backend/ontology/tools.py` | **内核**（含内核通用工具） | 复用内核 6 个工具（query_entity/create/update/traverse/execute/confirm + query_task/update_task） |
| `backend/main.py` | **内核** | 复用。tools/skills/prompt 从注册表聚合 |
| `backend/ontology/store.ttl` | **vertical**（clearance） | clearance 专属；新 vertical 建自己的 TTL |
| `backend/ontology/actions/*.yaml` | **vertical**（clearance） | clearance 专属；新 vertical 建自己的 actions 目录 |
| `backend/verticals/<name>/` | **vertical** | **新 vertical 在此新建**：config.py / tools.py / skills/ / 本体 / 种子 |
| `backend/models/schemas.py` | **vertical**（clearance） | 当前是 clearance 的 Pydantic 镜像；新 vertical 建自己的 schemas（见 02 Phase D 说明） |
| `backend/skills/store-ontology/` | **vertical**（clearance） | clearance 专属；新 vertical 建自己的 skills 目录 |
| `data/*.json` | **vertical**（clearance 种子） | 新 vertical 的种子放 `data/<vertical>/*.json`（见命名约定） |

**判断口诀**：如果一段代码提到具体的领域名词（NearExpiryProduct / clearance / 折扣 / 出清），它是 vertical；如果只认 `VerticalConfig` 和 registry，它是 kernel。

## 3. 命名与目录约定

### 3.1 vertical 标识

- vertical 名：`snake_case`，如 `clearance` / `equipment_repair` / `inventory_count`。
- 目录：`backend/verticals/<vertical_name>/`，内含 `config.py`（注册入口）。
- Python 包：`backend/verticals/__init__.py` 已存在；每个 vertical 子目录也需 `__init__.py`。

### 3.2 TTL 命名空间 prefix

- 每个 vertical 用自己的 prefix，从 TTL `@prefix <name>: <...> .` 行读取（parser 动态解析）。
- clearance 用 `store:`（历史遗留，已工作）；新 vertical 建议用 vertical 名，如 `repair:`。
- prefix 只需在该 vertical 的 TTL 内一致；不影响其它 vertical。

### 3.3 存储文件隔离

- **每个 vertical 的种子数据放独立子目录**：`data/<vertical_name>/*.json`。
- TTL 的 `store:storage "<file>.json"` 写文件名；`VerticalConfig.data_dir` 指向 `data/<vertical_name>`。
- clearance 历史数据在 `data/`（未迁子目录，保留作回归基线）；新 vertical 一律用子目录。
- 多 vertical 共享的实体（如门店组织 Region/Store/Employee）—— MVP 各 vertical 自带副本；共享层留 v2。

### 3.4 Action / Skill 命名

- Action api_name：`snake_case` 动词短语，全 vertical 命名空间**建议加 vertical 前缀**避免冲突（如 `create_repair_ticket` 而非 `create_ticket`）。
- Skill 目录：`backend/verticals/<name>/skills/<skill_name>/SKILL.md`。

## 4. 什么时候该建 vertical，什么时候不该

**应当建 vertical**：一组业务实体有独立的生命周期、受治理的写操作（Action）、跨多步的工作流。如：设备维修工单、库存盘点、会员退换货。

**不应当建 vertical**（用现有工具即可）：一次性查询、纯读取报表、单条 CRUD。这些用内核 `query_entity` / `create_entity`（降级 CRUD）即可，不必建模成 vertical。

判断见业务本体建模规范 §3.4（何时建新 Object Type）与 §5.2（何时建 Action Type）—— vertical 是"一组 Object + Action + 工作流"的集合，粒度比单个 Object 大。
